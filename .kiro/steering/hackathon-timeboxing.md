# Hackathon - Timeboxing y Gestión del Tiempo

## Regla de oro
Si algo toma más de 30 minutos sin progreso visible, hay que pivotar o simplificar.

## Distribución sugerida del tiempo (hackathon de 24h)

| Fase | Tiempo | Actividad |
|------|--------|-----------|
| Ideación | 1-2h | Definir problema, solución, y alcance del MVP |
| Setup | 1h | Scaffold, repo, herramientas |
| Core features | 10-12h | Desarrollo de funcionalidades principales |
| Integración | 2-3h | Conectar todo, datos reales, flujo completo |
| Polish | 2-3h | UI, UX, edge cases críticos |
| Demo prep | 1-2h | Pitch, guión, ensayo, video backup |
| Buffer | 2-3h | Imprevistos, bugs, descanso |

## Para hackathons más cortas (8-12h)
- Ideación: 30 min
- Setup: 30 min
- Core: 4-6h
- Integración + Polish: 1-2h
- Demo prep: 1h

## Señales de alerta
- "Solo un fix más y queda" → Probablemente no. Evaluar si es crítico.
- "Esto debería ser fácil" → Timebox de 15 min. Si no sale, buscar alternativa.
- "Necesitamos refactorizar primero" → No. Hacer la feature sobre lo que hay.
- "Falta agregar [feature nice-to-have]" → ¿Es necesaria para la demo? Si no, skip.

## Priorización de features
Usar la matriz de impacto vs esfuerzo:
- Alto impacto + Bajo esfuerzo → HACER PRIMERO
- Alto impacto + Alto esfuerzo → Hacer si hay tiempo
- Bajo impacto + Bajo esfuerzo → Hacer si sobra tiempo
- Bajo impacto + Alto esfuerzo → NO HACER
